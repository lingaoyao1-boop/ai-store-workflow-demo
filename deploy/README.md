# 连锁门店 AI 业务 SOP 工作流 Demo

这是一个用于求职作品集的本地演示项目，主题为“连锁饮品门店智能订货与库存预警”。

## 它解决什么问题

门店每天需要根据历史销量、库存、在途数量和保质期决定补多少货。纯人工容易漏看数据；完全交给大模型又不可控。本项目采用：

1. 本地规则引擎完成所有数字计算。
2. DeepSeek只解释结果、生成行动清单和SOP文本。
3. 超过审批阈值或出现临期风险时必须人工确认。

页面中的“计算说明”栏目展示了当前补货公式、计算示例、人工审批阈值，以及真实应用升级为时间序列预测时需要收集的数据。

## 手动录入数据

进入“数据录入”，每款饮品填写17个数字：

- 过去14天每日销量：14个数字。
- 当前库存：1个数字。
- 在途库存：1个数字。
- 最近批次距离到期天数：1个数字。

点击“保存并重新预测”后，系统会把数据保存在当前浏览器的本地存储中，并立即更新今日补货建议。点击“恢复该饮品示例”可以还原演示数据。

选择14天是为了分别计算前7天和最近7天，判断销量趋势。正式企业版本建议导入28-90天数据，并增加星期、天气、节假日和促销信息。

## 配置 DeepSeek

DeepSeek按输入和输出Token计费。规则计算、页面浏览和数据表格不消耗Token，只有点击“生成 AI 建议”才调用一次API。

推荐方式：点击页面右上角的“API 设置”，输入DeepSeek API Key。密钥只保存在当前页面内存中，刷新或关闭页面后自动清除，不写入文件、`localStorage`或服务端日志。

也可以使用环境变量：

```powershell
$env:DEEPSEEK_API_KEY="你的DeepSeek API Key"
python server.py
```

API Key只保存在启动服务的环境变量中，不会写入网页文件。

## 发布到互联网

推荐使用 GitHub + Render：

1. 在GitHub创建一个新仓库。
2. 将本目录中的全部文件上传到仓库根目录。
3. 登录Render，选择 `New > Web Service`，连接该GitHub仓库。
4. 如果Render识别到 `render.yaml`，按配置创建服务；也可以手动设置：
   - Runtime：Python 3
   - Build Command：留空
   - Start Command：`python server.py`
   - 环境变量：`HOST=0.0.0.0`
5. 部署完成后，Render会生成一个公开的HTTPS网址，将该网址发送给别人即可。

不要在GitHub或Render中配置你个人的DeepSeek Key。每位访问者点击页面右上角的“API 设置”，填写自己的Key。密钥只在当前页面内存中保存，并通过HTTPS发送给本项目后端，再转发给DeepSeek。

注意：部署者控制服务器，因此访问者应只在自己信任的部署地址中填写API Key。公开发布时建议保留完整源码，让使用者能够检查密钥处理逻辑。

## 面试讲法

“我没有让大模型直接决定补货数量，因为企业硬规则必须稳定、可审计。系统先根据历史销量、库存、保质期和审批阈值完成确定性计算，再把精简后的异常结果交给DeepSeek生成店长行动建议。这样既控制Token成本，也降低幻觉影响。”
